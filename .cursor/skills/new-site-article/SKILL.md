# Новая статья на сайте (taraswww777.dev)

## Когда применять

Пользователь просит **новую статью**, **опубликовать черновик**, **добавить материал в блог** или доработать существующий MDX в `content/articles`.

## Обязательные шаги

1. **Ориентир по формату** — открыть 1–2 соседние статьи в `content/articles/*.mdx`, чтобы совпали импорты, тон и структура блоков.
2. **Имя файла и slug** — `content/articles/YYYY-MM-DD-краткий-kebab-slug.mdx`. Дата публикации в имени и в контенте должны быть согласованы (или явно обсудить с пользователем).
3. **Шаблон MDX** — в начале файла:

   - `import { MdxTemplate } from 'src/components/mdx';`
   - `import MdxLayout from 'src/app/MdxLayout';`
   - при необходимости (как в других статьях) — `import { LINKS } from 'src/constants/links';`

   Обёртка:

   ```tsx
   <MdxLayout>
     <MdxTemplate
       pubdate={'YYYY-MM-DD'}
       title={'…'}
       ogCanonicalUrl={ARTICLE.link}
       description={'…'}
     >
   ```

   **`ARTICLE` не объявлять** — подставляется при сборке из манифеста (`src/lib/mdxUtils.ts`).

4. **Манифест** — в `content/pages-manifest.json` в объект `articles` добавить запись с тем же ключом, что и slug (без `.mdx`):

   - `slug`, `title` (как у статьи или согласованный заголовок для списков),
   - `status`: `published` или `draft`,
   - `publishedAt`: дата `YYYY-MM-DD` для опубликованных; для черновиков можно `""` по аналогии с существующими записями.

5. **URL** — страница будет доступна как `/articles/{slug}`.

## Редакторская доработка текста

- Дать **вводный абзац** и явные правила/ограничения, если в черновике они размыты (например: нет MR между хостингами, но канон для `main`/`develop` в другом месте).
- Убрать **двусмысленность «источник истины»** для веток и процессов; при появлении второго направления синхронизации — отдельный подзаголовок, предупреждения про расхождения и `--force-with-lease`.
- Сохранить **практические блоки** черновика (команды, шпаргалки, таблицы), но **сшить в один поток** без дублирования мысли.
- Для узких терминов и аббревиатур (**GPG**, протоколы и т.д.) — **одна** внятная ссылка на определение (например, русскоязычная Википедия), без лишних дублей ссылок подряд.
- Если меняется смысл или акцент статьи — обновить **`title` и `description`** в MDX и **`title`** в манифесте. **Slug не менять** без запроса пользователя (стабильность URL).

## Черновик на диске

Файл-источник (например, `two-repo.md` в корне) **не удалять и не перезаписывать** без явной просьбы.

## Проверка

- Убедиться, что slug в имени файла, в манифесте и в ссылках согласован.
- Прогнать линтер/сборку по возможности; если среда требует интерактива — проверить диагностики IDE по новому MDX.

## Дополнительно

Детали подстановки `ARTICLE` / `ARTICLES` — в `src/lib/mdxUtils.ts`.
