# Issue #8 — Связка статей с обсуждениями VK и Boosty

## 1. Контекст

Сайт статичный, поэтому комментарии как встроенная динамика внутри проекта не подходят как основной источник обсуждения.
Источник истины по комментариям и обсуждениям — внешние площадки:

- VK (личная страница)
- Boosty (личный аккаунт)

Решение: привязывать каждую статью к соответствующим постам в соцсетях и отображать ссылки на обсуждение в статье.  
Дополнительно (best effort): встроить VK-виджет обсуждения, если это не ломает статику.

## 2. Цель задачи

Сделать MVP-процесс, который:

1. Берет опубликованные статьи из `content/pages-manifest.json`.
2. Создает (или подготавливает) посты в VK/Boosty через локально запускаемый скрипт.
3. Сохраняет связь `article -> social post` обратно в `pages-manifest.json`.
4. Исключает дубли при повторных запусках.
5. Показывает ссылки на обсуждение на странице статьи.

## 3. Область (scope)

### Входит в MVP

- Только сущность `articles`.
- Локальный запуск скрипта публикации.
- Режим `dry-run`.
- Идемпотентность по `slug`.
- Отдельное поле планирования по МСК (ISO `+03:00`) вне `publishedAt`.
- Логи локального запуска в `logs/pub-posts/<slug>.log.md`.
- Настройка корневого хоста через env с возможностью override параметром запуска.
- Блок ссылок на VK/Boosty на странице статьи.
- Попытка добавить VK-виджет клиентским способом (best effort, без hard dependency).

### Не входит в MVP

- Обязательный CI-режим публикации (может быть добавлен позже).
- Полноценные автоматические ретраи (повтор через ручной перезапуск).
- Медиа-вложения.
- Сложный шаблонизатор текста поста.

## 4. Ключевые требования

### 4.1 Данные и связка

Связка хранится в существующем `content/pages-manifest.json`.

Расширение записи статьи (концепт):

- `social.scheduleAt` — дата/время публикации в ISO с `+03:00`, отдельно от `publishedAt`.
- `social.lastSyncAt` — служебная метка последней синхронизации.
- `social.vk`:
  - `postId`
  - `url`
  - `status` (`pending | scheduled | published | failed | skipped`)
  - `publishedAt`
- `social.boosty`:
  - `postId`
  - `url`
  - `status` (`pending | scheduled | published | failed | skipped`)
  - `publishedAt`

### 4.2 Идемпотентность

- Уникальный ключ: `article.slug`.
- Если у статьи по площадке уже есть `postId` или `url` — повторный запуск создает статус `skipped` и не создает дубликат.
- Повторный запуск допустим как ручной ретрай для неуспешных кейсов.

### 4.3 Запуск и окружение

- Основной режим запуска: локальный.
- Параметры скрипта:
  - `--dry-run`
  - `--host` (override host)
  - `--only-slug` (точечный запуск)
- Корневой host по умолчанию читается из env (`SITE_URL`), текущий рабочий: `https://taraswww777.github.io/taraswww777.dev/`.
- Секреты и токены: `.env` локально и CI env (git env secrets).

### 4.4 Контент поста (текущий шаблон)

Базовый шаблон:

`Ура, у меня вышла новая статья: <Название>. <Ссылка>`

Без медиа. Ограничения и формат по площадкам учитываются отдельно (VK/Boosty).

## 5. План реализации

## Этап 1. Подготовка модели манифеста

1. Обновить типы манифеста (`src/lib/pagesManifestShared.ts`) под `social`.
2. Убедиться, что чтение/запись манифеста (`src/lib/pagesManifest.ts`) корректно работает с новыми полями.
3. Добавить безопасную обратную совместимость для старых записей без `social`.

Результат этапа: манифест способен хранить связи с соцсетями и статус синхронизации.

## Этап 2. CLI-скрипт публикации (локальный)

1. Добавить скрипт публикации в `scripts/` (например, `publish-social-posts.js|ts`).
2. Добавить npm-команды в `package.json`:
   - `social:publish`
   - `social:publish:dry`
3. Реализовать сценарий:
   - загрузка манифеста;
   - выборка `articles` со `status = published`;
   - фильтрация по `--only-slug` (если указан);
   - построение URL статьи от заданного host;
   - проверка дублей;
   - вызов адаптеров VK/Boosty;
   - обновление манифеста и запись логов.
4. Для `dry-run`:
   - не выполнять внешние API-вызовы;
   - писать план действий в лог.

Результат этапа: рабочий локальный оркестратор синхронизации.

## Этап 3. Адаптеры внешних площадок

1. Выделить интеграционный слой (например, `src/lib/social/*`) с единым контрактом:
   - `createOrSchedulePost(input): { postId, url, status, publishedAt }`.
2. VK:
   - таргет — личная страница;
   - токен и настройки из env;
   - обработка ошибок и частичных отказов.
3. Boosty:
   - аналогичный интерфейс;
   - уточнение доступных API-возможностей по месту;
   - при необходимости fallback-ветка (минимум: фиксируем failed + лог причины).

Результат этапа: скрипт работает через унифицированные клиенты API.

## Этап 4. UI статьи: ссылки и виджет

1. На странице статьи (`src/pages/articles/[slug].tsx`) добавить блок обсуждений:
   - ссылка на VK при наличии `social.vk.url`;
   - ссылка на Boosty при наличии `social.boosty.url`.
2. Попробовать встроить VK-виджет обсуждения клиентским способом:
   - без влияния на SSG;
   - без падения страницы при недоступности VK-скрипта;
   - graceful degradation до обычных ссылок.

Результат этапа: пользователь статьи видит каналы обсуждения.

## Этап 5. Логи, диагностика и проверка

1. Формат логов: `logs/pub-posts/<slug>.log.md`.
2. Каждый прогон логирует:
   - входные параметры (`dry-run`, host, only-slug);
   - действия по площадкам;
   - итоги (`published/scheduled/skipped/failed`);
   - причины ошибок.
3. Проверить сценарии приемки.

Результат этапа: процесс прозрачен и воспроизводим.

## 6. Критерии готовности (Definition of Done)

1. Для опубликованной статьи можно запустить локальный сценарий связки с VK/Boosty.
2. В `pages-manifest.json` сохраняются `postId/url/status/date` по площадкам.
3. Повторный запуск не создает дублей и помечает кейсы как `skipped`.
4. `dry-run` не вызывает внешние API, но показывает полный план действий в логах.
5. На странице статьи отображаются ссылки на обсуждение.
6. Если VK-виджет недоступен, страница не ломается, ссылки продолжают работать.
7. Корневой host задается через env и может быть переопределен флагом.

## 7. Риски и заметки

- Ограничения VK API для личной страницы могут повлиять на отложенные публикации.
- Возможности Boosty API по create/schedule нужно валидировать в процессе внедрения.
- Для CI-публикации отдельно проверить возможность безопасного push обновленного манифеста.

## 8. Следующий статус после старта работ

После начала реализации изменить `status`:

- `created` -> `in_progress`

После завершения и прохождения критериев:

- `in_progress` -> `done`
